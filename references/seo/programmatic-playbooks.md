# Programmatic SEO Playbooks

## Overview

Programmatic SEO generates hundreds or thousands of pages from structured data and templates. Each page targets a specific long-tail keyword, and together they create massive organic traffic.

**Key principle:** Every page must provide unique value. Template text alone is not enough — each page needs data, context, or insight specific to its topic.

## Playbook 1: Location Pages

**Pattern:** [Service] in [City/State/Country]

**Example:** "Podcast hosting in Austin", "Web design in London"

### When to Use
- Local service businesses or marketplaces
- SaaS with location-specific data
- Any business where location modifies the offering

### Template Structure
```
/[service]/[city-state]/

H1: [Service] in [City, State]
- Local market data (population, businesses, industry stats)
- Service description customized for local context
- Local case studies or testimonials
- Pricing (if location-specific)
- Local competitors or alternatives
- FAQ with location-specific answers
- CTA: Get started in [City]
```

### Unique Content Sources
- Census/demographic data per location
- Local business statistics
- Regional pricing differences
- Local customer testimonials
- Area-specific regulations or requirements

### Scale Potential
- US cities (30,000+ population): ~1,000 pages
- US + Canada + UK: ~3,000 pages
- Global: 10,000+ pages

## Playbook 2: Integration Pages

**Pattern:** [Your Product] + [Other Product] Integration

**Example:** "Zapier + Slack integration", "HubSpot + Salesforce"

### When to Use
- SaaS products with integrations or API connections
- Marketplace platforms
- Any product that works with other tools

### Template Structure
```
/integrations/[other-product]/

H1: Connect [Your Product] with [Other Product]
- What you can do with this integration
- Step-by-step setup guide
- Use cases (3-5 specific workflows)
- Screenshots or diagrams
- Alternative approaches
- FAQ
- CTA: Try the integration free
```

### Unique Content Sources
- Actual integration capabilities and features
- Specific workflow examples
- User-reported use cases
- Comparison with manual alternatives
- Setup complexity and time

### Scale Potential
- Based on integration catalog: 50-500+ pages

## Playbook 3: Comparison Pages

**Pattern:** [Product A] vs [Product B]

**Example:** "Notion vs Asana", "Mailchimp vs ConvertKit"

### When to Use
- Competitive markets with many alternatives
- Products where buyers actively compare options
- High commercial intent keywords

### Template Structure
```
/compare/[product-a]-vs-[product-b]/

H1: [Product A] vs [Product B]: [Year] Comparison
- Quick verdict (who each is best for)
- Feature comparison table
- Pricing comparison
- Pros/cons for each
- Use case recommendations
- Migration considerations
- FAQ
- CTA: Try [Your Product] free
```

### Unique Content Sources
- Actual feature differences (research required)
- Pricing data (keep updated)
- User reviews and sentiment
- Specific use case recommendations
- Your unique perspective as a player in the market

### Scale Potential
- Top competitors × each other: 50-200 pages

## Quality Control Checklist

Before scaling any playbook:

### Pre-Launch (Pilot: 10-20 pages)
- [ ] Each page has 500+ words of unique content
- [ ] At least 60% of content is unique per page (not shared template)
- [ ] Each page targets a distinct search query
- [ ] Pages have unique title tags and meta descriptions
- [ ] Internal linking structure connects pages to hubs
- [ ] No thin content that adds no value beyond the template
- [ ] Mobile-friendly layout
- [ ] Page load time under 3 seconds

### Post-Pilot Validation
- [ ] Pages are being indexed (check GSC)
- [ ] Pages are generating impressions within 30 days
- [ ] No manual actions or quality warnings in GSC
- [ ] Bounce rate is acceptable (under 70%)
- [ ] Average time on page is reasonable (over 30 seconds)

### Scaling Checklist
- [ ] Data pipeline is reliable and maintainable
- [ ] Content quality is consistent across all pages
- [ ] Monitoring in place for broken pages or missing data
- [ ] Update schedule defined (how often data refreshes)
- [ ] Deindex strategy for pages with no data or zero search volume

## Internal Linking Strategy

### Hub-and-Spoke Model
```
Category Hub (/[category]/)
  ├── Spoke 1 (/[category]/[item-1]/)
  ├── Spoke 2 (/[category]/[item-2]/)
  ├── Spoke 3 (/[category]/[item-3]/)
  └── ... (hundreds of spokes)
```

### Linking Rules
- Every spoke links UP to its hub
- Every spoke links ACROSS to 3-8 related spokes
- Hub links DOWN to top-performing spokes
- Blog content links TO relevant programmatic pages
- Programmatic pages link TO relevant blog content
- Avoid orphan pages (every page has inbound links)
